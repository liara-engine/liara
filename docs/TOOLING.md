# Tooling

> The complete tooling reference for Liara: how the project is built,
> tested, validated, packaged, and published. This document describes
> the **mechanics** of the project; for the **why** behind these
> choices, see [`ARCHITECTURE.md`](ARCHITECTURE.md). For the
> day-to-day **workflow**, see [`CONTRIBUTING.md`](CONTRIBUTING.md).

## 2. Build System

### CMake with Presets

The project uses CMake 3.29 or newer, configured exclusively through
presets. Each module repository contains a `CMakePresets.json` file
defining configurations for the supported platform/compiler/build-type
combinations. There is no support for "configure CMake by passing
options on the command line"; the presets are the supported
interface.

The base preset structure mirrors what was used in the previous
engine, with simplifications:

```json
{
  "version": 6,
  "configurePresets": [
    { "name": "base", "hidden": true, ... },
    { "name": "linux-base", "inherits": "base", "hidden": true, ... },
    { "name": "windows-base", "inherits": "base", "hidden": true, ... },
    { "name": "linux-debug-gcc", "inherits": "linux-base", ... },
    { "name": "linux-debug-clang", "inherits": "linux-base", ... },
    { "name": "linux-release-gcc", "inherits": "linux-base", ... },
    { "name": "linux-release-clang", "inherits": "linux-base", ... },
    { "name": "windows-debug", "inherits": "windows-base", ... },
    { "name": "windows-release", "inherits": "windows-base", ... }
  ],
  "buildPresets": [...],
  "testPresets": [...]
}
```

Each non-hidden preset is invokable from the command line and used
by the CI. The CI does not pass extra options to CMake; it invokes
the preset and the preset is fully sufficient.

### Single-Config Generator on Linux, Multi-Config on Windows

On Linux, the Ninja generator is used with single-config builds: a
debug build and a release build are separate CMake configurations.
On Windows, the Visual Studio generator is used with multi-config
builds: one configuration directory holds both debug and release.

This asymmetry reflects platform conventions: Linux developers
expect to choose configuration at configure time, Windows developers
expect to choose at build time. Fighting either expectation produces
friction.

### Out-of-Source Builds

Build artifacts go to `build/<preset-name>/`, not to the source
directory. The source tree contains no build output. CMake refuses
in-source builds with an explicit error message.

### Compile Commands Database

CMake generates `compile_commands.json` for every Linux build. The
file is symlinked to the workspace root by the bootstrap script, so
that clangd (consumed by VSCode, Neovim, CLion, and other editors)
finds it automatically.

This file is also the foundation for clang-tidy, code coverage tools,
and SonarCloud's analysis. It is never committed to the repository
but must always be available locally.

### Modules-Optional Build

The previous engine experimented with C++20 modules. This experiment
showed that GCC's module support remains unstable in 2026, while
MSVC and Clang are usable. To avoid having a build that fails on
some compilers, the new project follows a different strategy:

- The reference build uses traditional headers, on all platforms.
- A separate experimental preset (`linux-release-clang-modules`,
  `windows-release-modules`) builds with modules where supported.
- The CI runs the reference build on all platforms but runs the
  modules-experimental build only on the platforms where it is known
  to work.

This avoids the situation where every contributor needs to debug
GCC's module bugs to land a small fix. Modules can be re-evaluated
when toolchain support is uniform.

## 3. Dependency Management

### vcpkg in Manifest Mode

External dependencies are managed through vcpkg in manifest mode.
Each module repository contains a `vcpkg.json` declaring its
dependencies and a `vcpkg-configuration.json` pinning the registry
baseline.

When CMake configures a project, it invokes vcpkg, which reads the
manifest and ensures the declared dependencies are available. The
first configuration on a fresh machine downloads and builds the
dependencies; subsequent configurations use cached binaries.

### Why vcpkg

Three options were considered: vcpkg, Conan, and using each
distribution's package manager. The third option is rejected because
it produces inconsistent builds across distributions (Ubuntu vs Arch
vs Fedora ship different versions of the same library) and because
it does not work on Windows without significant friction.

vcpkg is preferred over Conan for this project because:
- Microsoft maintains it, which means MSVC builds are first-class.
- Manifest mode integrates cleanly with CMake's `find_package` flow.
- The triplet system (specifying target architecture, build type,
  static/dynamic linking) is more explicit than Conan's profiles.
- The default registry is large enough to cover everything the
  project needs.

The cost of vcpkg is the initial dependency build, which can be slow
on first configuration. This is mitigated by the binary cache.

### Binary Cache

vcpkg's binary caching is enabled in CI through GitHub Actions cache
and locally through a per-developer cache directory. Cache keys
include the manifest hash, the triplet, and the compiler version, so
cache invalidation happens automatically when any of these change.

The cache makes "fresh CI runs" approximately as fast as
"incremental builds", which is essential for the project's
contribute-friendly philosophy: a contributor who waits 20 minutes
for CI to validate a typo fix loses interest.

### Submodules and FetchContent: Forbidden by Default

Git submodules and `FetchContent` are not used for external
dependencies. Submodules complicate the developer's local workflow
(easy to forget to update them, conflicts on detached HEADs);
`FetchContent` re-downloads sources on every fresh build, which is
the slow path that vcpkg's binary cache exists to avoid.

The exception: `FetchContent` is allowed for **header-only**
dependencies that are not in vcpkg's registry, where the cost of
re-downloading a few kilobytes of headers is negligible. This
exception is approved on a case-by-case basis through the
introduction of a dedicated ADR.

### Dependency List by Module

Each module's `vcpkg.json` declares only that module's direct
dependencies. The current expected list:

**`liara-interfaces`**: no dependencies.

**`liara-core`**: doctest (test only), tomlplusplus, miniaudio,
cgltf or tinygltf (TBD), stb (header-only), glm (internal
implementation, not exposed), spdlog (or own logger; TBD).

**`liara-renderer`**: doctest (test only), Vulkan-Headers,
vulkan-memory-allocator, glslang or shaderc (for runtime shader
compilation if needed), imgui[vulkan-binding,sdl3-binding].

The Vulkan SDK itself is **not** managed by vcpkg; it is a system
dependency that the developer installs (instructions in
[`BOOTSTRAP.md`](BOOTSTRAP.md)).

### Inter-Module Dependencies

Dependencies between Liara's own modules are not handled by vcpkg.
They are handled differently depending on whether the build is a
workspace build (all modules together) or a consumer build (one
module installed, another consuming it):

- **Workspace build**: the meta repository's superbuild
  CMakeLists.txt uses `add_subdirectory()` to include each module.
  Inter-module dependencies are resolved as targets in the same
  build tree.
- **Consumer build**: each module installs CMake config files via
  `install(EXPORT ...)`, and consumers use
  `find_package(LiaraInterfaces REQUIRED)` etc.

Both flows are tested in CI, on every push.

## 4. Reusable Workflows

The `.github` organization-level repository contains reusable GitHub
Actions workflows that every module repository invokes. This avoids
duplicating CI logic across N repositories.

### Why Reusable Workflows

The alternatives are:

- **Copy-paste** the workflow YAML across repositories. Maintenance
  nightmare; a fix in one repo's CI does not propagate.
- **Composite actions**. Better than copy-paste, but limited in what
  they can do (no matrix, no jobs).
- **Reusable workflows**. Full workflow capabilities, called from
  consuming repos with parameters. The right tool for this scale.

### Workflow Catalog

The `.github` repo contains workflows under
`.github/workflows/reusable-*.yml`. The current catalog:

**`reusable-build-and-test.yml`**: the canonical build-and-test
matrix. Inputs: which repository to build (the caller passes its
own name), CMake preset to use, whether to run tests, whether to
collect coverage. Outputs: build artifacts, test results, coverage
files.

**`reusable-lint.yml`**: clang-format and clang-tidy checks. Inputs:
repository name, source directories to scan. Fails if formatting is
incorrect or if clang-tidy reports new warnings.

**`reusable-docs.yml`**: Doxygen generation, integration with
`docs-shared` navbar, and gh-pages deployment. Inputs: repository
name, target directory in gh-pages (`dev`, `vX.Y.Z`).

**`reusable-release.yml`**: release-please integration. Inputs:
repository name, release-type (one of `cmake-cpp`, `simple`).
Produces release PRs and, when merged, tags releases.

**`reusable-sonarcloud.yml`**: SonarCloud analysis. Inputs:
repository name, SonarCloud project key. Requires a token configured
as an organization secret.

Each consuming repository has a small workflow file
(`.github/workflows/ci.yml`) that calls these reusable workflows.
The consuming workflows are short, typically under 30 lines.

### Versioning the Reusable Workflows

The reusable workflows are versioned. Consuming workflows reference
them by tag (e.g., `liara-engine/.github/.github/workflows/reusable-build-and-test.yml@v1`).
A breaking change to a reusable workflow is a tag bump, and consumers
update their reference at their own pace.

This protects against the situation where a fix to the CI infrastructure
silently breaks every module's CI on every push.

## 5. CI Pipelines

Each module repository has its own CI pipeline in
`.github/workflows/ci.yml`. The pipeline calls the reusable workflows
for the actual work; the per-module file just configures the inputs.

### Triggers

The CI runs on:

- Every push to `main`.
- Every pull request targeting `main`.
- Every push to a tag matching `vX.Y.Z` (for release builds).
- Manual dispatch through the GitHub UI (for debugging).

The CI does **not** run on pushes to feature branches that are not
yet associated with a PR. This avoids burning compute on
work-in-progress that the developer is not ready to validate.

### Build Matrix

For each PR or push, the build matrix is:

| OS              | Compiler    | Configuration | Notes                  |
|-----------------|-------------|---------------|------------------------|
| Ubuntu 24.04    | GCC 14      | Debug         | Tests + coverage       |
| Ubuntu 24.04    | GCC 14      | Release       | Performance build      |
| Ubuntu 24.04    | Clang 18    | Debug         | Tests, no coverage     |
| Ubuntu 24.04    | Clang 18    | Release       | Performance build      |
| Windows Server 2022 | MSVC 2022 | Debug       | Tests + coverage       |
| Windows Server 2022 | MSVC 2022 | Release     | Performance build      |

Six configurations. All must pass for the CI to be considered green.

The matrix uses Ubuntu rather than Arch because GitHub-hosted runners
do not offer Arch and self-hosted Arch runners would add operational
burden. The promise that "the project supports Arch" is made by the
fact that **the developer uses Arch daily**; the CI tests Ubuntu as a
reasonable proxy for "modern Linux". A scheduled weekly job verifies
the build on an Arch container to catch divergence.

### Parallelism and Caching

Each matrix job runs independently. Caches are used aggressively:

- **vcpkg binary cache**: shared across matrix entries with the same
  triplet. Hit rate is typically >95% on PRs that don't touch
  dependencies.
- **ccache** (Linux) and **sccache** (Windows): compile cache for
  source files, dramatically speeds up incremental CI builds. Cache
  is keyed by source content, so it survives across PRs.
- **Doxygen cache**: not currently planned, but considered if doc
  generation becomes a bottleneck.

The total CI time per push, with warm caches, is under 5 minutes per
matrix entry, with all entries running in parallel: end-to-end
under 10 minutes.

### Required Status Checks

The following checks are required to merge a PR:

- All build matrix entries pass.
- clang-format produces no diff.
- clang-tidy produces no new warnings.
- doctest tests all pass.
- Coverage does not decrease (with a configurable tolerance).
- SonarCloud quality gate passes.

Documentation generation is not a required check (a doc-only failure
should not block code merges); it runs but only reports.

## 6. Testing

### doctest

The project uses doctest as its test framework. The choice over
Catch2 is motivated by compile times: doctest is roughly an order of
magnitude faster to compile, and at this project's scale that
difference compounds across every CI run.

Tests are organized as separate executables per module, located in
each module's `tests/` directory. Each test executable is a doctest
runner, registered with CTest:

```cmake
add_executable(liara_core_tests tests/main.cpp tests/test_ecs.cpp ...)
target_link_libraries(liara_core_tests PRIVATE doctest::doctest liara_core)
add_test(NAME liara_core_tests COMMAND liara_core_tests)
```

This integration with CTest means that the same `ctest` invocation
runs all tests across all modules, both locally and in CI.

### Test Categories

Tests are tagged by category, allowing selective execution:

- `[unit]`: pure unit tests, no I/O, no Vulkan, no OS interaction.
  Run on every CI build.
- `[integration]`: tests that exercise multiple modules together,
  may use the file system or load real assets. Run on every CI
  build but excluded from the developer's "fast feedback" run.
- `[gpu]`: tests that require a Vulkan device. Run only in CI
  configurations that have GPU access (currently none on
  GitHub-hosted runners; this category is reserved for future
  self-hosted runners).
- `[benchmark]`: performance benchmarks. Run on every CI build
  with results recorded but not gating.
- `[slow]`: tests that take more than a few hundred milliseconds.
  Excluded by default to keep the fast feedback loop fast.

A developer running tests locally typically runs `[unit]` only; CI
runs everything except `[gpu]`.

### Test Discipline

- Tests live in the module they test. A test that exercises code
  from multiple modules lives in the module that initiated the
  feature being tested.
- Test files are named `test_*.cpp` and follow the same code style
  as production code.
- Tests are written alongside features, not retrofitted. A PR that
  adds a feature without tests is not merge-ready.
- Tests use the public C interfaces of `liara-interfaces` for
  integration-style tests, and may use C++ implementation details
  for module-internal unit tests.

### Coverage

Coverage is collected on Linux GCC Debug builds via gcovr, producing
a Cobertura XML report. On Windows, OpenCppCoverage produces an
equivalent report.

The coverage reports are uploaded to Codecov, which provides
historical tracking and PR comments showing changes in coverage.

The project does not target a specific coverage percentage. The
target is **direction**: each PR should not decrease coverage
without justification. This pragmatic approach avoids both the
"100% coverage cult" (which produces tests that test the test
framework) and the "coverage doesn't matter" excuse.

The coverage badge on the README links to Codecov. The exact
percentage shown is the average across all modules.

## 7. Static Analysis

### clang-format

Code formatting is enforced by clang-format 18+. The configuration
lives in `.clang-format` at the root of each repository. The
configuration is identical across repositories and is reviewed for
changes via the org-level `.github` repo's templates.

The CI runs `clang-format --dry-run --Werror` on every C++ source
file. Any deviation fails the build. There is no "format on save and
commit" optional pre-hook; the CI is the source of truth.

The detailed style is documented in
[`CODE_STYLE.md`](CODE_STYLE.md).

### clang-tidy

clang-tidy provides linting beyond formatting: catching common bugs,
enforcing modern C++ idioms, flagging anti-patterns. The
configuration lives in `.clang-tidy` at the root of each repository.

clang-tidy runs as part of every CI build, parallelized across
source files. The result is filtered to **new warnings introduced by
the PR**: existing warnings in untouched code are reported but not
required to be fixed in unrelated PRs. This avoids the situation
where every PR has to clean up legacy warnings before being
mergeable.

The clang-tidy configuration uses a curated set of checks rather
than `*` (all checks). The selected checks are listed in
`CODE_STYLE.md` with rationale.

### SonarCloud

SonarCloud provides a third layer of analysis: bugs, code smells,
duplication, security hotspots, and overall maintainability metrics.
It complements clang-tidy by tracking trends over time and providing
a web UI for exploration.

SonarCloud is free for open-source projects. Each module repository
is a separate SonarCloud project, and the meta repository displays
aggregate badges in its README.

The SonarCloud quality gate is configured to fail the CI if:
- New code coverage falls below 80%.
- New code introduces critical bugs.
- New code introduces security vulnerabilities.
- New code's reliability rating is below A.

Existing-code metrics are not gated; only new code (the diff in the
PR) is judged. This makes SonarCloud's feedback actionable without
becoming a blocker on every PR.

## 8. Conventional Commits

The project uses Conventional Commits 1.0. Commit messages follow
the pattern:

```
<type>(<scope>): <subject>

<body>

<footer>
```

The header is mandatory; the body and footer are optional. Lines
are wrapped at 72 characters.

### Types

The accepted types are:

- `feat`: a new feature visible to consumers.
- `fix`: a bug fix.
- `docs`: documentation changes only.
- `style`: code style changes (formatting, missing semicolons; not
  used much because clang-format makes most of these unnecessary).
- `refactor`: code changes that are neither features nor fixes.
- `perf`: performance improvement.
- `test`: adding or modifying tests.
- `build`: changes to build system, vcpkg, CMake.
- `ci`: changes to CI configuration.
- `chore`: routine maintenance.
- `revert`: reverts a previous commit.

### Scopes

Scopes identify the area of the project a commit affects. In a
multi-repo layout, scopes correspond to subsystems within a module:

In `liara-core`:
- `ecs`, `math`, `assets`, `logger`, `settings`, `events`, `loop`,
  `platform`.

In `liara-renderer`:
- `device`, `swapchain`, `pipeline`, `passes`, `targets`, `imgui`,
  `debug`.

In `liara-interfaces`:
- `core`, `renderer`, `physics`, `version`, `types`.

In `liara` (meta):
- `launcher`, `docs`, `packaging`, `scripts`.

Cross-cutting scopes available everywhere: `build`, `ci`, `deps`.

### Breaking Changes

A breaking change is signaled in two ways: the type is suffixed with
`!` (e.g., `feat(renderer)!: change packet structure`) **and** the
footer contains a `BREAKING CHANGE:` block describing the impact.
Both signals are required.

For modules that expose interfaces (most of them), a breaking change
in the conventional-commits sense triggers a major version bump
through release-please.

### Validation

Conventional commit format is validated by commitlint, run in CI as
part of the lint workflow. Non-conforming commits fail the PR's
checks. There is no Husky pre-commit hook because Husky requires
Node.js in every developer's environment, which adds friction.

## 9. Release Automation

### release-please

Each repository is configured with release-please, Google's release
automation tool. The configuration lives in
`release-please-config.json` at the repo root, with a manifest at
`.release-please-manifest.json` tracking the current version.

release-please works by maintaining a "release PR" that accumulates
changes from conventional commits. When the release PR is merged, a
new version is tagged, a changelog entry is created, and a GitHub
release is produced.

Each repository has its own independent release cycle. There is no
synchronization between modules' versions.

### Changelog Generation

release-please generates `CHANGELOG.md` automatically from
conventional commits. The format follows Keep a Changelog. Each
release section lists features, fixes, and breaking changes,
grouped by scope.

The generated changelog is reviewed in the release PR before
merging. Manual edits are possible but rare; the discipline of
writing good commit messages produces good changelog entries
automatically.

### Compatibility Matrix Updates

When a release happens in any module, the meta repository's
`compatibility.toml` may need updating. This is **not** automated;
the developer manually adds an entry when promoting a combination of
module versions to "tested together".

This explicit step exists because not every release is meant to be
rolled into a meta repository release. A patch release of
`liara-renderer` may or may not warrant a new entry in the matrix,
and the human is the right judge of that.

## 10. Documentation Generation

### Per-Module API Documentation: Doxygen

Each module repository generates API documentation with Doxygen. The
configuration lives in `Doxyfile` at the repo root, customized to:

- Process only public headers and intentionally documented source
  files.
- Use the templates from `docs-shared` for consistent navigation
  (the navbar described in `MODULES.md` section 8).
- Generate HTML to `build/docs/`.
- Include a search index.
- Cross-reference symbols across modules where relevant.

The generated HTML is published to the module's `gh-pages` branch
under a subdirectory named after the version (`dev/`, `v0.1.0/`,
etc.). The reusable docs workflow handles the publication.

### User Documentation: mdBook

User-facing documentation (tutorials, guides, conceptual articles)
lives in the meta repository under `docs/user/`, written in
Markdown and built with mdBook.

User documentation covers:

- Getting started: install, build, run the sample game.
- Concepts: ECS, render packets, asset management, settings,
  modularity.
- Tutorials: building a small scene, handling input, playing
  sounds, writing a simple game.
- Reference: configuration options, environment variables,
  command-line flags.

The mdBook output is published to the meta repository's `gh-pages`
branch under `user/`.

### The Documentation Hub

The meta repository's `gh-pages` branch hosts the documentation
hub at `liara-engine.github.io`. The hub:

- Displays a landing page introducing the engine.
- Provides links to the user documentation (mdBook).
- Links to per-module API documentation, with the version dropdown
  consuming `version.json`.
- Links to GitHub repositories.

The hub uses the navbar from `docs-shared`, ensuring consistency
with the per-module docs. The navbar's version dropdown updates
URLs across all linked documentation simultaneously.

### `version.json`

The `version.json` file at the root of the hub lists available
versions and the URLs for each module's documentation at each
version:

```json
{
  "current_dev": "dev",
  "versions": [
    {
      "label": "dev",
      "modules": {
        "core": "https://liara-engine.github.io/liara-core/dev/",
        "renderer": "https://liara-engine.github.io/liara-renderer/dev/",
        "interfaces": "https://liara-engine.github.io/liara-interfaces/dev/"
      }
    },
    {
      "label": "v1.0.0",
      "modules": {
        "core": "https://liara-engine.github.io/liara-core/v1.0.0/",
        "renderer": "https://liara-engine.github.io/liara-renderer/v1.0.0/",
        "interfaces": "https://liara-engine.github.io/liara-interfaces/v1.0.0/"
      }
    }
  ]
}
```

This file is updated by the release process: when a meta release
happens, the `version.json` gains a new entry. The previous `dev`
entry continues to point to the latest development docs.

### Coverage Badges and Reports

Coverage reports from Codecov are accessible from each module's
README via badges. The hub aggregates coverage information from all
modules into a single page, generated by a small script that polls
the Codecov API.

## 11. GitHub Pages Deployment

### Per-Module gh-pages Branch

Each module repository has a `gh-pages` branch that hosts its API
documentation. The branch is automatically populated by the
reusable docs workflow:

```
liara-core/gh-pages/
├── dev/                    # Current development docs
│   ├── index.html
│   └── ...
├── v0.1.0/                 # Released version docs
│   └── ...
├── v0.2.0/
│   └── ...
└── ...
```

The deployment is non-destructive: pushing new docs to `dev/` does
not touch `v0.1.0/`. Tagged releases produce a new versioned
subdirectory by copying `dev/` at the time of the tag.

### Branch Protection

The `gh-pages` branches do not have branch protection. They are
written to exclusively by CI; manual writes are not expected.

### Custom Domain

The hub's URL is `liara-engine.github.io`, the default GitHub Pages
URL for an organization. A custom domain is not configured; doing
so would add operational burden (DNS management, certificate
renewal) for marginal benefit.

## 12. Branch Protection and Required Reviews

### Main Branch

Each module repository's `main` branch is protected:

- Direct pushes are forbidden.
- All changes go through pull requests.
- All required CI checks must pass before merge.
- Force pushes are forbidden.
- Branch deletion is forbidden.

The "required reviews" rule is set to **zero** for solo development:
the developer reviews their own PRs, the CI is the gatekeeper. If
the project gains contributors, this is revisited and at least one
review is required.

### Squash Merge Only

Pull requests are merged via squash-and-merge. This produces clean
linear history on `main`, with each PR appearing as one commit. The
commit message of the squash uses the PR's title (which itself
follows conventional commits) and the PR description as the body.

This avoids the noise of "fix typo", "respond to review comment"
commits cluttering the history, while preserving the information in
the PR's discussion.

### Tag Protection

Tags matching `v*.*.*` are protected. Only release-please can create
them, through its merge-the-release-PR mechanism.

## 13. Local Development Workflow

### Workspace Setup

The recommended local setup:

1. Clone the meta repository.
2. Run `./scripts/setup-workspace.sh`. This clones every module
   repository into `workspace/`, configures vcpkg, and produces a
   superbuild CMakeLists.txt.
3. Open `workspace/` in the editor of choice (CLion, VSCode, Neovim
   with clangd).
4. Configure CMake with a preset of choice
   (`linux-debug-clang` is recommended for development).

The workspace produces a single `compile_commands.json` covering all
modules, which clangd consumes for cross-module navigation,
completion, and refactoring.

### Pre-Commit Checks

The project does not require pre-commit hooks. The CI is the source
of truth, and developers are expected to run the CI checks locally
before pushing.

A helper script, `./scripts/check.sh`, runs the same checks the CI
runs (clang-format, clang-tidy, tests, build) on the current
workspace. Running this before pushing avoids the "trivial CI
failure on PR" cycle.

### Compile Cache

ccache (Linux) is configured automatically by the CMake presets
when found in PATH. On Windows, sccache is used. These caches are
local to each developer's machine and are not shared across
developers; the CI has its own caches.

### Editor Integration

The project supports any editor that consumes `compile_commands.json`:

- **CLion**: opens the workspace `CMakeLists.txt` directly.
- **VSCode**: with the C/C++ extension or clangd extension.
- **Neovim**: with clangd via nvim-lspconfig.
- **Vim**: with vim-lsp or coc.nvim configured for clangd.

The `.clang-format` and `.clang-tidy` files at each repo root drive
the editor's formatting and linting. No editor-specific
configuration is committed to the project.

## 14. Performance Tracking

### Benchmark Results in CI

The benchmark suite (`[benchmark]` test category) runs in CI on
every push to `main`. Results are logged as artifacts and, longer
term, plotted on a dashboard so that performance regressions are
visible.

The dashboard is **not yet built** in v0.x. The initial plan is to
log results as JSON files attached to CI runs and to write a small
script that fetches and plots them on demand. A proper dashboard
(perhaps deployed alongside the docs hub) is deferred to v1.x.

### Continuous Profiling

No continuous profiling is set up in v0.x. Profile data is captured
manually when a developer notices a performance issue. The
`[benchmark]` tests serve as a coarse early warning: a 2x
regression in a benchmark is visible in the CI logs.

## 15. What Is Explicitly Not Used

To avoid the temptation of adding tools "just because they exist":

- **No conan**: vcpkg fills the role.
- **No bazel, meson, premake, or other build systems**: CMake is
  sufficient.
- **No Husky or pre-commit hooks framework**: CI checks are the
  source of truth.
- **No Jenkins, GitLab CI, CircleCI**: GitHub Actions is sufficient
  and is where the code lives.
- **No proprietary code analysis tools** beyond SonarCloud's free
  tier.
- **No package signing infrastructure** (deferred until there is a
  real need).
- **No Docker for the runtime** (the engine is not containerized;
  Docker is used only for CI image consistency, not for shipping).

Each "not used" entry is the result of an explicit decision. If a
new contributor proposes adding one of these, they should reference
the relevant section of this document and explain what changed.
